The Rutter Pattern: AI Memory Through Continuous Refinement

December 21, 2025 AI WorkflowsClaude Code

“A rutter was a small book containing the detailed observation of a pilot who had been there before. It recorded magnetic compass courses between ports and capes… the how we got there and how we got back: how many days on a special tack, the pattern of the wind… where there were friends and where foes.” — James Clavell, Shōgun

In the age of sail, a rutter wasn’t the voyage log. It was the extracted wisdom that made the next voyage safer.

The Pattern

A rutter wasn’t everything the pilot experienced. It was what the pilot chose to record for the next voyage:

  • Safe passages and dangerous shoals
  • Where friends and foes could be found
  • How many days on a specific tack
  • The pattern of the wind

Your AI memory system should work the same way: distilled navigation for future sessions.

Accumulate, Then Synthesize

The rutter emerges from accumulated experience, not from individual session summaries.

Here’s the pattern I follow:

  1. Many sessions accumulate — raw voyages, experiments, mistakes, wins
  2. Park application synthesizes — once enough voyages exist, extract the durable patterns
  3. Future sessions boot from the rutter — start with distilled wisdom, not empty context

The key insight: you can’t write a rutter after one voyage. You need enough experience to see what patterns hold up.

After accumulating enough sessions, I run a synthesis pass:

The Rutter vs. The Voyage Log

Voyage Log Rutter
Everything that happened What matters for next time
Chronological Organized by utility
Gets longer over time Gets refined over time
Raw data Distilled wisdom

The voyage log asks: “What did we do?” The rutter asks: “What should the next pilot know?”

What Doesn’t Make the Cut

A voyage log might include:

  • “Tried three different API endpoints before finding the right one”
  • “Spent 40 minutes debugging a typo in the config file”
  • “Asked clarifying questions about the requirements”
  • “Refactored the approach twice before settling on the final design”
  • “The user seemed frustrated around message 47”

The rutter keeps:

  • “Use /api/v2/export, not /api/v2/highlights — the latter doesn’t include favorites”
  • “Config keys are case-sensitive in production but not locally”
  • “Always confirm scope before implementation on ambiguous requests”

The voyage was interesting. The rutter is useful.

When to Write the Rutter

Not after every session. After enough sessions to see patterns:

  • When the same mistake happens twice
  • When a workaround becomes standard practice
  • When you notice yourself explaining the same thing to fresh contexts
  • When a technique proves itself across multiple situations

The rutter captures what held up, not what seemed good at the time.

The Implementation

  1. Start sessions by reading the rutter — Don’t start cold. Load the distilled context from previous voyages.

  2. Accumulate voyages, then synthesize — Individual sessions build up raw material. Periodically run a park pass to extract the patterns worth keeping.

  3. Refine, don’t append — The rutter gets better over time by removing what didn’t hold up, not by adding everything new.

  4. Test against fresh contexts — A good rutter works when handed to a new agent with no prior history.

The Punchline

The pilots who survived the age of sail didn’t have better ships. They had better rutters.

Build the rutter. Let experience accumulate. Then distill.
